# 直播 Demo 规划

这个 demo 的目标不是只做一个可运行的直播页面，而是为后续演进成 React 组件提前做能力拆分和架构设计。

当前阶段先把直播链路跑通，并且把配置、页面、流能力拆开，避免后续重构成本过高。

## 项目目标

- 把当前直播 demo 逐步演进成可复用的 React 组件。
- 将配置加载、推流、播放、设备控制、桌面共享等能力拆分清楚。
- 让页面只负责展示和交互，业务逻辑尽量下沉到 provider、hook 和 service。
- 兼容后续接入更多流协议和更多业务场景。

## 核心设计原则

- 配置优先：组件加载时先读取配置，再初始化推流或播放能力。
- 页面与逻辑解耦：页面不直接承载复杂的流控制逻辑。
- 能力可复用：推流页和播放页都可以独立抽成组件。
- 协议可扩展：推流和播放都预留多协议接入能力。
- 先拆分，再优化：先保证架构清晰，再逐步补齐体验细节。

## Config Provider

后续会引入一个统一的 `Config Provider`，它负责存储和读取用户配置，是整个直播能力的入口。

它的职责包括：

- 在组件加载时拉取配置。
- 保存当前配置，并向下游组件提供读取能力。
- 根据配置决定推流和播放的初始化方式。
- 承载后续可能扩展的参数，例如房间信息、鉴权信息、默认设备、协议优先级和分辨率配置。

建议将配置分成两类：

- 静态配置：由业务系统、环境变量或外部注入提供。
- 动态配置：运行时从接口加载，或由用户在界面中修改后保存。

## 页面拆分规划

这个 demo 里实际会有两个独立页面，后续也会分别演进成对应组件。

### 推流页

推流页负责主播侧能力，目标是将 RTC 流推送到 OSS RS。

主要能力包括：

- 摄像头采集。
- 麦克风采集。
- 桌面共享。
- 窗口共享。
- 网页共享。
- 开始推流与停止推流。
- 推流状态、设备状态和错误状态展示。

需要注意的是，设备控制和流控制要拆开，不要把媒体采集逻辑直接耦合到页面里。

### 播放页

播放页负责观众侧能力，后续会支持两类播放方式：

- 低延迟 RTC 播放。
- 延迟更高但更省带宽和 CPU 的格式播放，例如 RTMP、HLS、RTSP。

播放页要根据配置和业务场景决定采用哪种协议，并保留后续扩展空间。

## 推流能力拆分

推流相关能力建议拆成以下几层：

- `Config Provider`：负责读取推流配置。
- `useMediaDevices`：负责摄像头、麦克风等设备枚举和切换。
- `useScreenShare`：负责桌面、窗口、网页共享。
- `usePublishStream`：负责真正的推流启动、停止和状态管理。
- `PublishPage`：只负责把这些能力组合成界面。

这样做的好处是，后续如果要把推流页变成独立组件，只需要复用这些能力层，而不是搬运整个页面。

## 播放能力拆分

播放相关能力建议拆成以下几层：

- `Config Provider`：负责读取播放配置和协议优先级。
- `usePlayStream`：负责 RTC、RTMP、HLS、RTSP 等播放方式的接入。
- `PlayPage`：只负责播放容器、状态展示和基础交互。

后续如果某个业务场景只需要 RTC 播放，也可以直接复用播放能力层，而不是依赖整页逻辑。

## 协议方向

推流侧当前目标是对接 OSS RS 的推流能力，后续实现时需要重点参考它的接入资料和能力约束。

当前 demo 的默认推流地址为：

- `webrtc://srs.idealjs.com/live/livestream`

这意味着主播侧推流链路默认会按照 WebRTC 推流来设计，后续如果需要支持更多推流目标，可以再把地址和协议抽象到配置中。

播放侧需要同时预留两条路线：

- RTC：用于低延迟播放场景。
- RTMP / HLS / RTSP：用于更高延迟但更省资源的播放场景。

后续具体接入时，可以由配置决定默认协议和降级策略。

## 设备能力要求

主播侧需要支持以下媒体能力：

- 网页分享。
- 窗口分享。
- 整个屏幕分享。
- 摄像头切换。
- 麦克风切换。

这些能力最好通过独立 hook 或 service 实现，避免页面层直接操作浏览器媒体 API。

## 后续演进方向

- 先完成配置加载和页面拆分。
- 再实现推流页的设备管理和桌面共享。
- 接着实现播放页的多协议接入。
- 最后把页面能力收敛成可复用 React 组件。

如果后续要正式组件化，还可以继续补充：

- 对外暴露的 props 和 events。
- 配置 schema。
- 错误码和状态码定义。
- 组件的受控和非受控模式。
- 服务端接口和权限模型。
